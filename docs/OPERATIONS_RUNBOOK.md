# Operations Runbook

This runbook explains how to inspect and maintain the repository without interrupting its scheduled automation.

## Daily Health Check

```bash
gh run list --repo P-r-e-m-i-u-m/daily-activity --limit 10
```

Healthy state:

- Recent scheduled workflows complete successfully.
- No repeated failed runs for the same workflow.
- The latest commits are on `main`.
- Docs generated by scheduled workflows land in the expected folders.

## Local Verification

```bash
npm install
npm test
npm audit
```

If local tests depend on unavailable services, inspect the failing test and prefer adding mocks or clearer docs instead of disabling coverage.

## Workflow Safety Rules

- Do not remove scheduled workflows unless the replacement is already working.
- Do not change secrets names without updating the workflow docs.
- Keep `cancel-in-progress: false` for workflows that write docs or releases.
- Prefer small workflow hardening changes over full rewrites.
- After workflow edits, run the affected workflow manually.

## Common GitHub Actions Checks

```bash
gh workflow list --repo P-r-e-m-i-u-m/daily-activity
gh run list --repo P-r-e-m-i-u-m/daily-activity --limit 20
gh run view <run-id> --repo P-r-e-m-i-u-m/daily-activity --log-failed
```

## Incident Response

When a workflow fails:

1. Read the failed job log.
2. Identify whether the problem is code, auth, GitHub runner availability, or a remote API failure.
3. Patch only the smallest affected workflow or script.
4. Trigger the workflow manually.
5. Delete stale failed runs only after a successful replacement run exists.
6. Record the fix in `CHANGELOG.md` or an incident note when the failure was meaningful.

## Release Hygiene

- Keep release automation idempotent.
- Retry remote GitHub calls that may return temporary 5xx errors.
- Check whether a release already exists before creating it.
- Make release notes readable for humans, not only automation.
